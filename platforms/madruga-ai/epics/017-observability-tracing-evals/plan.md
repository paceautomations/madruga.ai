# Implementation Plan: Observability, Tracing & Evals

**Branch**: `epic/madruga-ai/017-observability-tracing-evals` | **Date**: 2026-04-02 | **Spec**: `platforms/madruga-ai/epics/017-observability-tracing-evals/spec.md`
**Input**: Feature specification from `platforms/madruga-ai/epics/017-observability-tracing-evals/spec.md`

## Summary

Adicionar observabilidade end-to-end ao pipeline automatizado: tracing hierarquico (traces agrupando spans/pipeline_runs), captura de tokens/custo via parsing do JSON output do Claude CLI, eval scoring heuristico em 4 dimensoes, endpoints REST no daemon FastAPI, e pagina de dashboard no portal Astro com 4 tabs (Runs, Traces, Evals, Cost) atualizadas via polling 10s. Dados persistidos em SQLite estendendo o schema existente. Retention automatica de 90 dias com export CSV sob demanda.

## Technical Context

**Language/Version**: Python 3.11+ (backend), TypeScript/React (portal)
**Primary Dependencies**: sqlite3 (stdlib), structlog, FastAPI (daemon), React + @xyflow/react (portal existente), Astro Starlight
**Storage**: SQLite WAL mode (`.pipeline/madruga.db`) — novas tabelas `traces` e `eval_scores`, coluna `trace_id` em `pipeline_runs`
**Testing**: pytest (backend), manual (portal — sem test framework configurado para React islands)
**Target Platform**: Linux (WSL2 dev, systemd daemon em prod)
**Project Type**: Extensao de pipeline CLI + daemon web-service + portal static-site com React islands
**Performance Goals**: Dashboard carrega < 3s com 90 dias de dados. Overhead de observabilidade < 5% do tempo total do pipeline.
**Constraints**: Python stdlib + pyyaml apenas (sem deps novas para tracing). Scripts < 300 LOC cada. SQLite single-writer.
**Scale/Scope**: Single-user (1 operador), dezenas de runs/dia. ~1700 LOC total estimado (com boilerplate x1.5).

## Complexity Tracking

> Sem violacoes de constitution. Complexidade dentro do esperado.

| Aspecto | Decisao | Justificativa |
|---------|---------|---------------|
| Reutilizar `pipeline_runs` como spans | Sem tabela `spans` separada | Evita duplicacao. `pipeline_runs` ja tem todos os campos de span. Basta adicionar `trace_id` FK. |
| Eval scoring heuristico | Sem LLM calls para evals V1 | Zero custo adicional. Metricas quantitativas cobrem 80% do valor. Judge scores ingeridos quando node judge roda. |
| SVG puro para waterfall | Sem Recharts/D3 | ~50 LOC, zero deps, performance otima. Portal ja usa React islands. |
| Polling 10s | Sem SSE/WebSocket | Trivial em React, latencia aceitavel para monitoring humano. |

## Phase 0: Research (COMPLETE)

Documentado em `research.md`. 8 decisoes de pesquisa resolvidas:

| # | Decisao | Rationale |
|---|---------|-----------|
| R1 | Parse `claude -p --output-format json` para tokens/cost | Campos ja existem no schema, so precisam ser populados |
| R2 | Reutilizar `pipeline_runs` como spans (+ `trace_id` FK) | Evita duplicacao, reusa infraestrutura existente |
| R3 | `ALTER TABLE ADD COLUMN` seguro em SQLite | Operacao nativa, atomica, instantanea |
| R4 | Eval scoring heuristico (zero LLM cost) | 4 dimensoes com metricas quantitativas. Judge scores integrados quando existem. |
| R5 | Polling 10s (fetch periodico em React) | Trivial, tolerante a falhas, latencia aceitavel |
| R6 | Cleanup via 3 DELETEs sequenciais (sem CASCADE) | Mais explicito que CASCADE para FK adicionada via ALTER TABLE |
| R7 | SVG puro para waterfall (zero deps) | ~50 LOC, customizavel, performance otima |
| R8 | CORSMiddleware no daemon | Necessario para fetch cross-origin portal→daemon |

## Implementation Phases

### Fase 1: Schema + Captura Backend (Prioridade P1)

**Objetivo**: Persistir traces e metricas automaticamente a cada pipeline run.

1. **Migration 010_observability.sql** (~50 LOC)
   - CREATE TABLE traces
   - ALTER TABLE pipeline_runs ADD COLUMN trace_id
   - CREATE TABLE eval_scores
   - 7 indices
   - Aplica automaticamente via `migrate()` em db.py

2. **db.py: funcoes de trace/span/eval** (~150 LOC)
   - `create_trace(conn, platform_id, epic_id, mode, total_nodes) → trace_id`
   - `complete_trace(conn, trace_id, status)` — calcula agregados dos spans
   - `get_traces(conn, platform_id, limit, offset, status_filter) → list[dict]`
   - `get_trace_detail(conn, trace_id) → dict` (trace + spans)
   - `insert_eval_score(conn, trace_id, platform_id, epic_id, node_id, run_id, dimension, score, metadata)`
   - `get_eval_scores(conn, platform_id, node_id, limit) → list[dict]`
   - `get_stats(conn, platform_id, days) → list[dict]`
   - `cleanup_old_data(conn, days=90) → int` (retorna rows deletadas)

3. **dag_executor.py: tracing integration** (~100 LOC)
   - `parse_claude_output(stdout: str) → dict` — extrai tokens/cost do JSON
   - No `run_pipeline_async`: criar trace no inicio, associar a cada insert_run
   - No `dispatch_node_async`: capturar stdout, parsear JSON, retornar metricas
   - No `complete_run`: popular tokens_in/out/cost_usd/duration_ms
   - Na finalizacao: `complete_trace()` com status e agregados
   - Try/except em todo parsing — falha silenciosa (FR-011: best-effort)

### Fase 2: Eval Scoring (Prioridade P2)

4. **eval_scorer.py** (~150 LOC)
   - `score_node(platform_id, node_id, run_id, output_path, metrics) → list[EvalScore]`
   - 4 dimensoes heuristicas (R4):
     - `quality`: output nao vazio + sem marcadores de erro → 7.0 default. Se Judge score existe: normalizar.
     - `adherence_to_spec`: regex match de sections esperadas vs template
     - `completeness`: `min(10, actual_lines / expected_lines * 10)`
     - `cost_efficiency`: `10 - min(10, cost / avg_budget * 10)`, neutro 5.0 se sem historico
   - Chamado inline no dag_executor apos cada node completar com sucesso
   - Persistido via `insert_eval_score()`

### Fase 3: Daemon Endpoints (Prioridade P1)

5. **daemon.py: endpoints observabilidade** (~120 LOC)
   - CORSMiddleware com origins localhost
   - GET `/api/traces?platform_id=&limit=&offset=&status=`
   - GET `/api/traces/{trace_id}` (trace + spans + eval scores)
   - GET `/api/evals?platform_id=&node_id=&limit=`
   - GET `/api/stats?platform_id=&days=30`
   - GET `/api/export/csv?platform_id=&entity=traces|spans|evals&days=90`

6. **daemon.py: retention cleanup** (~30 LOC)
   - Periodic task no TaskGroup: `retention_cleanup(conn, shutdown_event, interval=86400)`
   - Chama `cleanup_old_data(conn, days=90)`
   - Log com structlog: rows deletadas por tabela

### Fase 4: Portal Frontend (Prioridade P2)

7. **observability.astro** (~20 LOC)
   - Nova pagina `/[platform]/observability`
   - Importa `ObservabilityDashboard` como React island com `client:load`

8. **ObservabilityDashboard.tsx** (~100 LOC)
   - State: activeTab, data per tab
   - Polling: `useEffect` com `setInterval(10000)` + `fetch()`
   - 4 tabs: Runs, Traces, Evals, Cost

9. **RunsTab.tsx** (~80 LOC)
   - Tabela de traces recentes (status, duracao, custo, nodes)
   - Status badges coloridos (running=blue, completed=green, failed=red)
   - Click em row → expande detalhes inline

10. **TracesTab.tsx** (~100 LOC)
    - Waterfall SVG: cada span como `<rect>` proporcional a duracao
    - Tooltip via `<title>` nativo
    - Color-coded por status
    - Seletor de trace (dropdown dos recentes)

11. **EvalsTab.tsx** (~80 LOC)
    - Scoreboard: tabela node × dimensao com scores coloridos
    - Trend sparklines (ultimos 5-10 runs) — SVG path simples
    - Destaque visual para scores < 5

12. **CostTab.tsx** (~80 LOC)
    - Bar chart SVG: custo por dia/semana
    - Tokens acumulados (in/out)
    - Top nodes por custo

### Fase 5: Testes (~200 LOC)

13. **test_db_observability.py** — CRUD traces, spans com trace_id, eval scores, cleanup, stats
14. **test_eval_scorer.py** — Scoring em cada dimensao, edge cases (output vazio, sem historico)
15. **test_daemon_observability.py** — Endpoints via httpx TestClient, CORS, paginacao
16. **test_observability_export.py** — CSV export (traces, spans, evals), headers, encoding
