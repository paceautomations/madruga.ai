# Implementation Plan: Commit Traceability

**Branch**: `epic/madruga-ai/023-commit-traceability` | **Date**: 2026-04-08 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `platforms/madruga-ai/epics/023-commit-traceability/spec.md`

## Summary

Link every git commit to an epic (or mark as ad-hoc). Provide full visibility of changes — past and future — in the SQLite DB and the Starlight portal. Core deliverables: `commits` table, post-commit hook, backfill script, portal Changes tab.

## Technical Context

**Language/Version**: Python 3.11 (hook + backfill), TypeScript/React (portal), SQL (migration)
**Primary Dependencies**: sqlite3 (stdlib), subprocess (git), json, React + Astro Starlight
**Storage**: SQLite WAL mode (`.pipeline/madruga.db`) — existing pattern per ADR-012
**Testing**: pytest (Python), manual (portal)
**Target Platform**: Linux (WSL2)
**Project Type**: CLI scripts + web dashboard component
**Performance Goals**: Hook < 100ms (non-blocking), backfill handles 1000+ commits
**Constraints**: Hook MUST NOT block commits (best-effort), backfill idempotent (INSERT OR IGNORE)
**Scale/Scope**: ~100-500 commits per platform, 3-5 platforms

## Architecture

### Data Flow

```
git commit → post-commit hook → hook_post_commit.py → commits table
                                                         ↑
backfill_commits.py (one-time) ─────────────────────────┘
                                                         ↓
portal ChangesTab.tsx ← commits-status.json ← platform_cli.py
```

### Key Decisions (from pitch.md)

1. **Composite SHA** for multi-platform commits (`sha:platform_id` uniqueness via INSERT per platform)
2. **No hard FK** on epic_id — epic may not exist in DB yet
3. **Best-effort hook** — catch all exceptions, never block git
4. **Branch-based epic detection** — parse `epic/<platform>/<NNN-slug>` from branch name
5. **Message tag override** — `[epic:NNN]` in commit message overrides branch detection
6. **Backfill uses git log** — walks full history, classifies by branch merge topology
7. **Source tracking** — `hook`, `backfill`, `manual` to distinguish provenance

## Implementation Phases

### Phase 1: Database Schema (migration 014)
- `commits` table with SHA, message, author, platform_id, epic_id, source, files_json
- Indexes on platform_id, epic_id, committed_at

### Phase 2: Post-Commit Hook
- `hook_post_commit.py` — reads HEAD commit metadata via `git log -1`
- Platform detection: scan changed files for `platforms/<name>/` prefixes
- Epic detection: parse branch name, check message tags
- Multi-platform: one row per platform touched
- Error handling: catch-all, log to stderr, exit 0 always

### Phase 3: Git Hook Wrapper
- `.specify/scripts/git-hooks/post-commit` — shell wrapper calling Python script
- Symlink/copy installation via setup script

### Phase 4: Backfill Script
- `backfill_commits.py` — walks `git log --all` for full history
- Classifies commits: epic branch commits, merge commits, direct-to-main (ad-hoc)
- Pre-006 heuristic: commits before epic branches existed get classified by file paths
- Idempotent: INSERT OR IGNORE on SHA unique constraint

### Phase 5: DB Integration
- Extend `db_pipeline.py` with `get_commits()`, `get_commits_by_epic()` helpers
- Wire into `post_save.py` reseed flow

### Phase 6: Portal Changes Tab
- `ChangesTab.tsx` — React component with table, filters (platform, epic, date range)
- Data source: `commits-status.json` generated by CLI export

### Phase 7: CLI Export
- `platform_cli.py export-commits` — generates JSON for portal consumption

### Phase 8: Testing
- Unit tests for hook (mock git subprocess)
- Unit tests for backfill (mock git log, verify classification)
- Integration: verify 103 commits in DB after backfill

## Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Hook slows down commits | Best-effort with catch-all + exit 0 |
| Backfill misclassifies pre-006 commits | Heuristic with file path matching, acceptable false positives |
| DB locked during hook | Retry once, then skip silently |
| Composite SHA breaks GitHub links | Accepted trade-off — SHA is for DB uniqueness, not URL linking |
